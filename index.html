01 / PROJECT OVERVIEW

从“看见数据”到“完成处置”

项目探索如何将分散的无人机设备、视频画面、AI 识别结果与应急流程整合进同一套工作台。

传统人工巡检存在范围大、周期长、危险区域难抵达的问题；而单独引入无人机后，又容易产生设备、视频、预警与任务相互割裂的新问题。设计目标是建立一个统一、可追踪、可协同的巡检闭环。

DESIGN CHALLENGE

三个核心设计问题

在没有真实业务系统约束的概念阶段，我将问题聚焦在信息整合、异常响应和跨角色协作三个方向。

01

信息分散

设备状态、巡检任务、视频流和异常记录分布在不同入口，管理者难以快速建立全局认知。

02

响应滞后

异常识别后缺少清晰的风险分级与处理状态，容易发生重复确认或无人跟进。

03

协同断层

设备管理员、值班人员与决策者关注不同信息，缺少统一的任务上下文与处置记录。

02 / USERS & INSIGHTS

围绕任务，而不是围绕功能

平台面对的不是单一操作员，而是不同信息颗粒度的多个角色。界面需要同时支持实时操作与管理决策。

KEY INSIGHTS

需求转化为设计策略

A

状态优先

列表中优先显示飞行、维护、风险和电量等影响下一步行动的状态信息。

B

逐层深入

首页呈现总览，模块页承载操作，详情与弹窗负责局部编辑，降低单页认知负担。

C

异常可追踪

预警不仅是一条消息，还应包含风险、位置、时间和处置状态，成为任务流的起点。

CORE USER FLOW

从 AI 发现异常到任务闭环

最关键的体验不是单个页面，而是异常如何在角色之间持续流转。

STEP 01执行巡检任务无人机按预设航线采集实时画面。
→
STEP 02AI 识别异常系统标记疑似人员、设施异常或信号问题。
→
STEP 03人工研判风险值班员确认风险等级、位置和可信度。
→
STEP 04创建处置任务指派负责人并补充任务要求。
→
STEP 05完成与复盘记录处置结果，数据进入统计分析。

06 / REFLECTION

项目复盘与下一步

这套作品强调的是产品结构与交互表达。若进入真实项目，还需要通过业务调研和数据验证进一步收敛。

“后台产品的价值不在于把所有数据都展示出来，而在于让不同角色知道：现在发生了什么，以及下一步应该做什么。”
先定义任务闭环页面结构应服务于业务流程，而不是简单罗列模块。
状态信息要可行动状态、风险和位置必须能直接支持下一步判断。
真实验证仍然重要下一阶段应访谈巡检人员，验证字段优先级和异常处置规则。
